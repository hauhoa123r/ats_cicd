1 What is an Interceptor

An interceptor is a small piece of code that sits in the middle of a request and a response. It can read or change the data while the data travels to its target, and it can also act on the data on the way back. Think of it as a checkpoint on a road. Every car that uses the road stops at the checkpoint, gets a quick check, and then keeps going.

Interceptors live on both sides of a web app. On the server side, Spring Boot gives you a HandlerInterceptor that runs before and after a controller handles a request. On the client side, Axios gives you request interceptors and response interceptors that run around every HTTP call the browser makes.

●

Key idea. An interceptor lets you run shared logic in one place, so you do not copy the same code into every controller or every API call.

2 Why we use Interceptors

Many tasks in an app are not part of the business logic, yet they need to happen on almost every request. These shared tasks are a great fit for interceptors. Here are common examples.

TaskWhat the interceptor does
LoggingRecord the method, the path, and the time taken for each request
Auth contextAttach a token or a credential before a call leaves the browser
Error handlingCatch a failed response in one place and react the same way every time
AuditingTrack who did what and when, without changing controller code
TimingMeasure how long a handler runs and flag slow requests

The main win is cleaner code. Without interceptors, the same logging line or the same token line would show up in many places. With interceptors, that logic stays in one file and applies everywhere.

3 How an Interceptor works

An interceptor wraps the normal flow of a request. The request enters the interceptor first, then moves on to the real handler, and finally the response passes back through the interceptor before it reaches the caller. The steps below show the order.

STEP 1The caller sends a request
STEP 2The interceptor checks or changes the request
STEP 3The handler builds a response
STEP 4The interceptor checks or changes the response
STEP 5The caller gets the final response

In Spring Boot the three hooks are named preHandle, postHandle, and afterCompletion. The method preHandle runs before the controller. The method postHandle runs after the controller but before the view is built. The method afterCompletion runs at the very end, even when an error happened.

4 Backend Interceptor in Spring Boot

To build an interceptor in Spring Boot, implement the HandlerInterceptor interface. The example below measures how long each request takes and writes a log line. It saves the start time on the request in preHandle and reads it back in afterCompletion.

LoggingInterceptor.java
package org.ats.common.interceptor;

import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Component;
import org.springframework.web.servlet.HandlerInterceptor;

@Component
@Slf4j
public class LoggingInterceptor implements HandlerInterceptor {

    private static final String START_TIME = "startTime";

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
        request.setAttribute(START_TIME, System.currentTimeMillis());
        log.info("Incoming request {} {}", request.getMethod(), request.getRequestURI());
        return true; // return true to continue, return false to stop the request
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) {
        long start = (Long) request.getAttribute(START_TIME);
        long duration = System.currentTimeMillis() - start;
        log.info("Completed {} {} with status {} in {} ms",
                request.getMethod(), request.getRequestURI(), response.getStatus(), duration);
    }
}

Once the class exists, register it so Spring applies it to the routes you pick. Create a config class that implements WebMvcConfigurer and add the interceptor to the registry. You can include some paths and skip others.

WebMvcConfig.java
package org.ats.config;

import lombok.RequiredArgsConstructor;
import org.ats.common.interceptor.LoggingInterceptor;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.config.annotation.InterceptorRegistry;
import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;

@Configuration
@RequiredArgsConstructor
public class WebMvcConfig implements WebMvcConfigurer {

    private final LoggingInterceptor loggingInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(loggingInterceptor)
                .addPathPatterns("/api/v1/**")
                .excludePathPatterns("/api/v1/auths/**");
    }
}
⚠

The return value matters. When preHandle returns true, the request moves on to the controller. When it returns false, the request stops and the controller never runs. This is how an interceptor can block traffic when a check fails.

5 Frontend Interceptor in Axios

Axios offers two kinds of interceptors. A request interceptor runs before the browser sends a call, and a response interceptor runs after the server replies. The example below builds on the shared Axios client already used in this project. The code uses plain function expressions so the flow is easy to read.

axiosClient.js
import axios from "axios";

const BASE_URL = "http://localhost:8080";

const axiosClient = axios.create({
  baseURL: BASE_URL,
  headers: { "Content-Type": "application/json" },
  withCredentials: true,
  timeout: 5000,
});

// Request interceptor: runs before every call leaves the browser
axiosClient.interceptors.request.use(
  function (config) {
    console.log("Calling", config.method, config.url);
    return config; // you must return config so the call can proceed
  },
  function (error) {
    return Promise.reject(error);
  }
);

// Response interceptor: runs after every reply arrives
axiosClient.interceptors.response.use(
  function (response) {
    return response; // pass a good response straight through
  },
  function (error) {
    if (error.response && error.response.status === 401) {
      // The session is missing or expired, so send the user to the login page
      window.location.href = "/login";
    }
    return Promise.reject(error);
  }
);

export default axiosClient;
⚠

Always return. A request interceptor must return the config object, and a response interceptor must return the response object. If you forget to return, the call gets stuck because the chain has nothing to pass along.

6 Applying it in this project

This project keeps the access token inside an httpOnly cookie. The browser sends that cookie on its own when withCredentials is true, so the frontend does not need a request interceptor to add a token by hand. The most useful frontend interceptor here is the response interceptor that reacts when a session expires.

The setup we suggest is the following.

  • On the frontend, add a response interceptor on the Axios client. When the server replies with status 401, first try to renew the session with a refresh token (see the next section), and only send the person to the login page if that renewal also fails. This keeps the user signed in without any interruption.
  • On the backend, add a LoggingInterceptor for every route under /api/v1. This gives a steady record of the method, the path, the status, and the time taken.
  • Keep the login check itself inside the existing JwtAuthenticationFilter. A filter is a better home for security than an interceptor, because a filter runs earlier in the chain. The next section explains the difference.

7 Refresh token with Axios

A good security setup gives the access token a short life, for example fifteen minutes. When it expires, the user should not be forced to log in again. Instead the app quietly asks the server for a new access token using a second, longer lived token called a refresh token. A response interceptor is the perfect place to run this renewal, because it can catch the 401 reply, fetch a fresh token, and replay the original call. The user sees nothing.

How the flow works

STEP 1A normal call returns 401 because the access token expired
STEP 2The interceptor calls the refresh endpoint, which reads the refresh cookie
STEP 3The server checks the refresh token, rotates it, and sets new cookies
STEP 4The interceptor replays the original call, now with a valid token
STEP 5If the refresh also fails, the app sends the user to the login page
●

Cookies carry the tokens. In this project both the access token and the refresh token are httpOnly cookies. The browser sends them on its own, so the interceptor never reads or writes a token by hand. It only needs to call the refresh endpoint and replay the original request.

Three problems to handle

A refresh interceptor looks simple, but three cases must be handled with care.

ProblemHow to solve it
Endless loop Mark the retried request with a flag such as _retry, and never run the refresh logic for the refresh call itself or for the login call.
Many calls fail at once If several requests get 401 together, refresh only once. Hold the other requests in a small queue and replay them after the single refresh finishes. This is called the single flight pattern.
Refresh itself fails The refresh token is expired or revoked. Clear the saved user and send the person to the login page.

Example interceptor

The example below uses the single flight pattern. While one refresh call is running, every other failed request waits in a queue. When the refresh succeeds, all waiting requests are replayed. When it fails, they are all rejected and the user is sent to the login page.

axiosClient.js
import axios from "axios";

const BASE_URL = "http://localhost:8080";

const axiosClient = axios.create({
  baseURL: BASE_URL,
  headers: { "Content-Type": "application/json" },
  withCredentials: true, // send the httpOnly access and refresh cookies
  timeout: 5000,
});

// Single flight state: one shared refresh call serves many failed requests
let isRefreshing = false;
let waitingQueue = []; // each item is { resolve, reject }

function flushQueue(error) {
  // When error is null the refresh worked, so wake up every waiting request.
  // Otherwise reject them all so the callers can handle the failure.
  waitingQueue.forEach(function (item) {
    if (error) {
      item.reject(error);
    } else {
      item.resolve();
    }
  });
  waitingQueue = [];
}

axiosClient.interceptors.response.use(
  function (response) {
    return response;
  },
  function (error) {
    const original = error.config;
    const status = error.response ? error.response.status : null;

    // Never refresh the auth calls themselves, or we could loop forever
    const isAuthCall =
      original.url.indexOf("/api/v1/auths/login") !== -1 ||
      original.url.indexOf("/api/v1/auths/refresh") !== -1;

    // Only react to 401, and only retry a request once
    if (status !== 401 || original._retry || isAuthCall) {
      return Promise.reject(error);
    }

    original._retry = true;

    // A refresh is already running: wait for it, then replay this request
    if (isRefreshing) {
      return new Promise(function (resolve, reject) {
        waitingQueue.push({
          resolve: function () {
            resolve(axiosClient(original));
          },
          reject: reject,
        });
      });
    }

    // This request starts the single refresh call
    isRefreshing = true;

    return axiosClient
      .post("/api/v1/auths/refresh")
      .then(function () {
        flushQueue(null);             // wake up the queued requests
        return axiosClient(original); // replay the request that started the refresh
      })
      .catch(function (refreshError) {
        flushQueue(refreshError);     // reject every waiting request
        window.location.href = "/login"; // refresh failed, so log out
        return Promise.reject(refreshError);
      })
      .finally(function () {
        isRefreshing = false;
      });
  }
);

export default axiosClient;
⚠

This is a teaching example. Keep it readable and adapt it to your own state handling. The important ideas are the same in every real version: react only to 401, refresh once for many requests, replay the waiting calls, and log out when the refresh fails.

What the backend provides

The frontend code above relies on a backend that issues a refresh token at login and rotates it on every refresh. In this project the server stores only the hash of each refresh token in the database, marks the old token as revoked on every rotation, and revokes all of a user's tokens if a used token appears again, which is a strong sign of theft. The endpoints are POST /api/v1/auths/login, POST /api/v1/auths/refresh, and POST /api/v1/auths/logout.

8 Interceptor and Filter

New developers often ask how an interceptor differs from a servlet filter, since both run around a request. This table makes the line clear.

PointFilterInterceptor
Belongs toThe servlet containerThe Spring MVC framework
Runs atThe outer edge, before Spring routes the requestInside Spring, around the controller
Knows the handlerNo, it sees the raw request and responseYes, it receives the handler object
Good forSecurity, encoding, the JWT checkLogging, auditing, timing, app level checks

In short, use a filter for low level needs such as security, and use an interceptor for needs that benefit from knowing which controller will handle the request.

9 Best practices

  • Keep each interceptor focused on one job, so it stays easy to read and to test.
  • Never put heavy or blocking work inside an interceptor, since it runs on every request.
  • On the backend, point the interceptor at the right routes with include and exclude patterns.
  • On the frontend, always return the config or the response so the chain keeps going.
  • Do not write secrets such as raw tokens into log lines.
  • Prefer a filter for security and an interceptor for logging and timing.

10 Summary

An interceptor is a checkpoint that runs around a request and a response. It is the right tool for shared tasks such as logging, timing, and session handling, because it keeps that logic in one place. Spring Boot gives you a HandlerInterceptor with the hooks preHandle, postHandle, and afterCompletion. Axios gives you request and response interceptors for the browser. In this project, a response interceptor renews the session with a rotating refresh token when the access token expires, so the user stays signed in without any interruption, while the security check stays inside the existing JWT filter.
